Fake Discount BG

Chrome Extension Live on Chrome Web Store ↗

"Bulgarian e-commerce websites inflate prices before events such as Black Friday and fake the discounts. Now my browser catches the manipulated prices."

The problem

During sale events, Bulgarian retailers (Emag, Ozone, Notino) raise prices days before the "discount" so the slashed price looks bigger than it is. There was no easy way to tell what was real.

What I built

A Chrome/Edge extension that auto-tracks every product I visit, giving a verdict directly on the product page: Fake discount / Real deal / Stable price / Still tracking — with reasoning text and an SVG price chart embedded into the page. It outclasses similar tools by performing background EAN extraction and cross-referencing prices via a backend Supabase table that automatically records data points when a user lands on a product. I also built a bot and data protection rules directly inside Supabase so scraping scripts cannot extract the database.

How I built it

Built using Claude Code. Site-specific content scripts sharing a common base class. Background service worker for message routing and analysis. Per-product Chrome local storage with O(1) reads and a write queue to prevent race conditions on rapid reloads.

Key Features

  • 3 marketplaces
  • EAN / Supabase cross-site check
  • Anti-bot data protection
  • Bilingual BG / EN
  • Keyboard shortcut
  • Price-target alerts
  • Export / import JSON
  • Persistent filters

The Orchestration Challenge Getting the AI to understand and respect content-script CSP isolation on strict sites like Ozone, while designing a robust background messaging architecture that handles multi-day data-collection without race conditions.

Flagship Project

Nova Control: Concrete Plant Dispatcher

Python Desktop App · 14 modules · 496 passing tests

"The most ambitious system I've built — a Windows desktop app reading live data from a real industrial PLC."

The problem

Bulgarian concrete plants run on Soviet-era dispatcher software with brittle UIs and no modern features. Operators are often elderly. They need a modern, monitor-only dispatcher that reads the existing Omron PLCs without ever risking a write to the live process — a single bad write to a concrete batching PLC could damage equipment or hurt people.

What I built

A Windows desktop application (.exe) that interfaces with Omron CJ1M-CPU11 / CJ2M-CPU32 PLCs over FINS / UDP, FINS / Serial, and Host Link — strictly READ-ONLY (the driver base class doesn't even expose write methods). 8 tabs covering Settings, Reports, Invoices, Production Management, Clients, SCADA Operator Screen, Requests, and Dispatcher (default). Recipe immutability for anti-fraud, hardware-bound licensing with offline grace, auto-database backups (180 retained), 3-tier polling (200 ms / 500 ms / 2000 ms) with staleness detection, full Bulgarian UI with high contrast and large fonts for elderly operators.

How I built it

Built with Cursor and Claude Opus. Python + PyQt6 + SQLite + a custom Omron driver layer. Architected as 14 independent modules with strict dependency boundaries enforced by Qt signal/slot communication. Current AI context windows aren't large enough to build a 20,000+ line application in one pass. To make it work, I use custom skills and detailed .md mapping files, allowing the AI to understand the entire module dependency graph and safely build a complex app module by module.

  • 14 modules built
  • 496 passing tests
  • Phase 4 of 5 in progress
  • Read-only at API level
  • Multi-plant profile system
  • Full Bulgarian operator UI

The Orchestration Challenge Managing a complex, multi-session state where the AI must respect safety contracts at the type and API level without hallucinations. It required rigorous prompting and the discipline to build an offline testing layer so the AI could iterate without touching the live PLC.

How I work

The orchestration is the skill.

I describe outcomes in plain English. I let AI translate intent into working code, then I review, test, and iterate. The core skill is orchestration: knowing what to ask, how to read a model's output critically, when to switch tools, and when a problem requires a different approach.

I don't just accept generated code. After verifying functionality manually, I rigorously prompt the AI to write comprehensive unit tests that validate backend logic and edge cases—not just syntax. Nova Control alone has 496 passing tests. The result is robust, tested software, not fragile AI slop.

My projects use real engineering practice: 3-branch deployments (alpha → beta → main), modular architecture with clear dependency boundaries, and .md mapping files that act as agent-handoff documents — so any new session picks up where the last one stopped without losing context.

I take ownership of the entire lifecycle — from architectural planning to deployment and bug fixing. The end result isn't just generated code; it's robust software that solves actual business needs.
