Fake Discount BG

Chrome Extension

"Bulgarian e-commerce inflates prices before Black Friday and fakes the discounts. Now my browser catches the manipulated prices."

The problem

During sale events, Bulgarian retailers (Emag, Ozone, Notino) raise prices days before the "discount" so the slashed price looks bigger than it is. There was no easy way to tell what was real.

What I built

A Chrome/Edge extension (Manifest V3) that auto-tracks every product I visit. After 7+ days of tracking, it gives a verdict directly on the product page: Fake discount / Real deal / Stable price / Still tracking — with reasoning text and an SVG price chart embedded into the page itself. Per-tab badges indicate fake discounts, real deals, and reached price targets.

How I built it

Built using Claude Code. Site-specific content scripts (Emag / Ozone / Notino) sharing a common base class. Background service worker for message routing and analysis. Per-product Chrome local storage with O(1) reads and a write queue to prevent race conditions on rapid reloads. EAN / GTIN extraction enables future cross-site matching.

  • 3 marketplaces
  • Bilingual BG / EN
  • Keyboard shortcut
  • Price-target alerts
  • Export / import JSON
  • Persistent filters

Taught me Manifest V3 service workers, content-script CSP isolation on strict sites (Ozone), and how to design for a multi-day data-collection window.

SIMA-MA Industrial Website

Bilingual Marketing Site

"A Bulgarian concrete-equipment manufacturer needed a real bilingual web presence — not just a Facebook page."

The problem

SIMA-MA Ltd. (Cherven Bryag, Bulgaria) builds concrete batching plants, valves, silo solutions, and custom steel fabrication. They needed a serious bilingual site to reach both Bulgarian and English-speaking buyers.

What I built

A 13-page bilingual website with directory-based i18n (/bg/ and /en/), a custom canvas particles animation themed to the industry (gravel / sand / cement), a 79-photo gallery with category accordions and lightbox, a fully styled EU-funded project page with 16 mandatory PDF documents, and a validated contact form.

How I built it

Architected with Claude Code and Codex. Static HTML + CSS + vanilla JS — no framework. Web3Forms for the contact form, Cloudflare R2 for image hosting (zero egress), Netlify deployment with a 3-branch strategy (alpha → beta → main).

  • 13 pages BG / EN
  • Custom canvas animation
  • ManufacturingBusiness JSON-LD
  • hreflang + sitemap lastmod
  • Skip-to-content + focus-visible
  • Responsive to 320px

Taught me directory-based i18n that actually scales, custom particle systems that pause when off-screen, and how to set up a real branch-based deployment pipeline.

Flagship Project

Nova Control: Concrete Plant Dispatcher

Python Desktop App · 14 modules · 496 passing tests

"The most ambitious system I've built — a Windows desktop app reading live data from a real industrial PLC."

The problem

Bulgarian concrete plants run on Soviet-era dispatcher software with brittle UIs and no modern features. Operators are often elderly. They need a modern, monitor-only dispatcher that reads the existing Omron PLCs without ever risking a write to the live process — a single bad write to a concrete batching PLC could damage equipment or hurt people.

What I built

A Windows desktop application (.exe) that interfaces with Omron CJ1M-CPU11 / CJ2M-CPU32 PLCs over FINS / UDP, FINS / Serial, and Host Link — strictly READ-ONLY (the driver base class doesn't even expose write methods). 8 tabs covering Settings, Reports, Invoices, Production Management, Clients, SCADA Operator Screen, Requests, and Dispatcher (default). Recipe immutability for anti-fraud, hardware-bound licensing with offline grace, auto-database backups (180 retained), 3-tier polling (200 ms / 500 ms / 2000 ms) with staleness detection, full Bulgarian UI with high contrast and large fonts for elderly operators.

How I built it

Built with Cursor and Claude Opus. Python + PyQt6 + SQLite + a custom Omron driver layer. Architected as 14 independent modules with strict dependency boundaries enforced by Qt signal/slot communication. Documented in long-form internal specs (a CLAUDE.md-style agent-handoff map) so any new AI session picks up where the last one left off — without losing context on the safety constraints or the module dependency graph.

  • 14 modules built
  • 496 passing tests
  • Phase 4 of 5 in progress
  • Read-only at API level
  • Multi-plant profile system
  • Full Bulgarian operator UI

Taught me long-form architecture documentation that survives across AI sessions, enforcement of safety contracts at the type / API level, and the discipline to not ship something safety-critical until the tooling is ready.
