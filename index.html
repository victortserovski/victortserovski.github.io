Fake Discount BG

Chrome Extension Live on Chrome Web Store ↗

"Built for the Bulgarian market, where discounts can be misleading. This extension tracks real price history and exposes when a promotion isn’t as good as it looks."

The problem

I built this extension for the Bulgarian market, where price tracking is especially difficult. Major retailers such as eMAG, Ozone, and Notino can change prices before sale periods, which makes discounts appear more significant than they really are. Existing price-tracking tools were not reliable enough for local stores, so I created a solution focused specifically on accurate tracking for Bulgarian shoppers.

What I built

A Chrome/Edge extension that analyzes every product I visit and shows a real-time verdict directly on the page: Fake discount, Real deal, Stable price, or Still tracking—complete with explanation and a visual price history chart.

For products with available EANs, it cross-checks prices through a backend system that builds shared price history across sites. When EAN data is missing—a common limitation in Bulgarian online stores—it continues tracking the product independently. The system also includes protections to secure the data from scraping.

How I built it

Built using Claude Code. I owned the product logic, testing strategy, and integration constraints — defining exactly how to extract prices across three different site structures, when to flag a fake discount, and how to store history reliably. My main challenge was validating the extension against strict anti-bot protections on sites like Ozone.bg, iterating until the background architecture handled multi-day data collection without breaking.

Key Features

  • 3 marketplaces
  • EAN / Supabase cross-site check
  • Anti-bot data protection
  • Bilingual BG / EN
  • Keyboard shortcut
  • Price-target alerts
  • Export / import JSON
  • Persistent filters

The Orchestration Challenge Testing and iterating against strict website security rules, and defining clear constraints so the AI could build a robust messaging architecture that handles multi-day data-collection without breaking.

Flagship Project

Nova Control: Concrete Plant Dispatcher

Python Desktop App · 14 modules · 496 passing tests

"The most ambitious system I've built — a Windows desktop app reading live data from a real industrial PLC."

The problem

Bulgarian concrete plants run on Soviet-era dispatcher software with brittle UIs and no modern features. Operators are often elderly. They need a modern, monitor-only dispatcher that reads the existing Omron PLCs without ever risking a write to the live process — a single bad write to a concrete batching PLC could damage equipment or hurt people.

What I built

A Windows desktop application (.exe) that interfaces with Omron CJ1M-CPU11 / CJ2M-CPU32 PLCs over FINS / UDP, FINS / Serial, and Host Link — strictly READ-ONLY (the driver base class doesn't even expose write methods). 8 tabs covering Settings, Reports, Invoices, Production Management, Clients, SCADA Operator Screen, Requests, and Dispatcher (default). Recipe immutability for anti-fraud, hardware-bound licensing with offline grace, auto-database backups (180 retained), 3-tier polling (200 ms / 500 ms / 2000 ms) with staleness detection, full Bulgarian UI with high contrast and large fonts for elderly operators.

How I built it

Built with Cursor and Claude Opus. A 20,000+ line application can't be built in one AI session — so I acted as the system architect. I wrote detailed .md mapping files defining 14 modules, their dependencies, and their safety constraints. Each module was built, tested offline against mock PLC data, and integrated incrementally — the same discipline you'd expect from a team of engineers, compressed into one person directing AI.

How I work
