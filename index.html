Fake Discount Bulgaria

Chrome Extension Live on Chrome Web Store ↗

"Covers 20 Bulgarian e-commerce sites. Tracks real price history automatically and exposes when a promotion isn’t as good as it looks."

The problem

Bulgarian e-commerce has a fake-discount problem. Retailers raise prices before a "sale" so the markdown looks bigger than it is. This happens across the entire market — from electronics giants like eMAG and Ozone to fashion stores like About You and pharmacies like Sopharmacy. Since no existing tool offered sufficient coverage across Bulgarian stores, this project was developed to support 20 of the country's major e-commerce websites.

What I built

A Chrome/Edge extension with 20 dedicated per-site content scripts that automatically tracks every product you visit. It shows a real-time verdict directly on the page — Fake discount, Real deal, Volatile price, Stable price, or Tracking — with a visual price-history chart and an explanation of why it reached that conclusion.

Every observation is also synced to a Supabase cloud database, building a shared price-history dataset across all installs. For products with exposed EAN barcodes, the system can cross-reference the same product across different stores. The extension handles single-page apps (About You, Answear, dm), currency traps (BGN vs EUR), out-of-stock detection, and upsell-rail isolation — each site has its own extraction logic.

How I built it

Built using Claude Code. The development involved establishing the core product architecture, verdict logic, and per-site extraction rules for all 20 stores—each presenting unique HTML structures, SPA navigation, currency variations, and lazy-loaded elements. The main orchestration challenge lay in specifying constraints clearly enough to produce 20 separate, robust content scripts feeding into a unified storage and verdict system, while surviving anti-bot protections without breaking.

Key Features

  • 20 Bulgarian e-stores
  • 5-tier verdict system
  • Supabase cloud sync
  • EAN cross-site matching
  • SPA navigation handling
  • Bilingual BG / EN
  • Price-target alerts
  • Export / import JSON

The Orchestration Challenge Defining extraction rules for 20 different site structures — SPAs, currency traps, anti-bot protections, out-of-stock edge cases — and specifying a unified verdict cascade that produces consistent, explainable results across all of them.

Flagship Project

Nova Control: Concrete Plant Dispatcher

Python Desktop App · 7 tabs · 1,637 tests · v2.0.0

"A workflow and recipe dispatcher for concrete batching plants, featuring a hardware-isolated safety architecture that ensures critical controls remain with the operator."

The problem

Concrete batching plants rely on dispatcher software to manage recipes, customer orders, and material tracking. However, the software in most local plants is old and clumsy. Because the software communicates directly with heavy machinery (scales, valves, and conveyors), there is a strict safety constraint: the dispatcher application must only write recipe values, and never trigger start, stop, or discharge commands. Those safety-critical controls must remain physically with the operator on the plant floor.

What I built

A Windows desktop application that integrates with a Delta PLC (with legacy serial communication support for older Omron systems). The software manages the entire dispatcher workflow across 7 specialized tabs: plant configuration, recipes, client tracking, active orders, live batch monitoring, real-time visualization, and automated PDF reporting.

The safety architecture is the core design achievement: all outgoing writes pass through a strict safety gate that blocks and logs any values outside allowed recipe ranges. Once recipe targets are pushed, the plant operator physically reviews them on the touchscreen and presses the physical START button to run the batch—keeping human oversight in control of the heavy machinery.

Additional features include a configurable Plant Builder, automated concrete passport PDF generation per truck, stock tracking with material substitution, and an active fuzzer engine to verify UI stability under randomized stress.

How I built it

The system was orchestrated using AI tools across 32 dedicated testing sessions to manage a codebase containing over 36,000 lines of code and 20,000 lines of test code. By consulting a concrete plant expert, 19 production decisions were mapped out covering how recipe adjustments, inventory deductions, and material substitutions behave in real-world scenarios.

To ensure industrial stability, the application was verified with 1,637 behavioural and integration tests. These cover cross-tab state communication, edge-case input validation, and randomized fuzzer runs, confirming the system operates reliably under extreme conditions.

Safety Architecture

PC pushes recipe targets only → Operator presses physical START → PC reads back status and logs results. The software cannot trigger start, stop, or discharge commands.
