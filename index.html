Fake Discount Bulgaria

Chrome Extension Live on Chrome Web Store ↗

"Covers 20 Bulgarian e-commerce sites. Tracks real price history automatically and exposes when a promotion isn’t as good as it looks."

The problem

Bulgarian e-commerce has a fake-discount problem. Retailers raise prices before a "sale" so the markdown looks bigger than it is. This happens across the entire market — from electronics giants like eMAG and Ozone to fashion stores like About You and pharmacies like Sopharmacy. No existing tool covered enough Bulgarian stores, so I built one that handles all 20 major sites.

What I built

A Chrome/Edge extension with 20 dedicated per-site content scripts that automatically tracks every product you visit. It shows a real-time verdict directly on the page — Fake discount, Real deal, Volatile price, Stable price, or Tracking — with a visual price-history chart and an explanation of why it reached that conclusion.

Every observation is also synced to a Supabase cloud database, building a shared price-history dataset across all installs. For products with exposed EAN barcodes, the system can cross-reference the same product across different stores. The extension handles single-page apps (About You, Answear, dm), currency traps (BGN vs EUR), out-of-stock detection, and upsell-rail isolation — each site has its own extraction logic.

How I built it

Built using Claude Code. I defined the product architecture, verdict logic, and per-site extraction rules for all 20 stores — each with different HTML structures, some using SPAs, some mixing currencies, some hiding prices behind lazy-load. My main challenge was specifying the constraints clearly enough that the AI could produce 20 separate content scripts that all feed into one unified storage and verdict system, while surviving anti-bot protections and SPA navigation without breaking.

Key Features

  • 20 Bulgarian e-stores
  • 5-tier verdict system
  • Supabase cloud sync
  • EAN cross-site matching
  • SPA navigation handling
  • Bilingual BG / EN
  • Price-target alerts
  • Export / import JSON

The Orchestration Challenge Defining extraction rules for 20 different site structures — SPAs, currency traps, anti-bot protections, out-of-stock edge cases — and specifying a unified verdict cascade that produces consistent, explainable results across all of them.

Flagship Project

Nova Control: Concrete Plant Dispatcher

Python Desktop App · 7 tabs · 1,637 tests · v2.0.0

"A full dispatcher system for a real concrete batching plant — managing orders, recipes, trucks, and live production, with a verified safety architecture that ensures only recipes ever reach the industrial controller."

The problem

Concrete batching plants need a dispatcher — someone who takes customer orders, assigns mixer trucks, sends the correct recipe to the plant's industrial controller, and tracks every kilogram of material used. Most Bulgarian plants either use outdated DOS-era software or manage everything on paper. The real challenge: this software talks directly to an industrial controller (PLC) that operates heavy machinery — scales, valves, conveyors, a mixer. The dispatcher needs to send recipes to the controller, but must never be able to start, stop, or override the physical production process. That responsibility belongs to the operator standing at the machine.

What I built

A Windows desktop application that connects to a Delta TP70P-32TP1R industrial controller over three communication channels — Ethernet, serial cable, or legacy RS-232 for older Omron plants — all in one application. The software handles the entire dispatcher workflow across 7 specialized tabs: plant configuration, recipe and inventory management, client and site tracking, incoming orders, expedition dispatch with live controller status, a real-time plant visualization screen, and comprehensive reporting with printable concrete quality certificates.

The safety architecture is the core achievement: the software can only write recipe data to the controller — never start, stop, or discharge commands. Every outgoing write passes through a safety gate that verifies it falls within the allowed recipe range. Any write outside that range is blocked, logged, and refused entirely. After the recipe arrives, the operator reviews it on the controller's touchscreen, adjusts water based on material humidity, and presses START physically — keeping a human in control of the actual production at all times.

Beyond the core dispatch workflow, the system includes a configurable Plant Builder (describe any plant layout — how many cement silos, aggregate bunkers, water tanks, additive dispensers, and mixers, along with their controller addresses), immutable recipe versioning (editing a recipe creates a new version while archiving the old one — critical for quality assurance and anti-fraud in construction), automatic concrete passport PDFs generated per truck with full batch details, material substitution tracking when stock runs out, hardware-bound licensing with remote management, and a dispatcher fuzzer that runs 1,000 randomized operator actions to find interaction bugs no manual testing could catch.

How I built it

This project demonstrates what AI orchestration looks like at serious scale. I wrote a 36,000+ word master specification — a complete document covering every business rule, safety constraint, controller protocol detail, and interface requirement. I conducted in-depth interviews with a concrete plant expert and documented 19 confirmed business decisions that govern how recipes, inventory, substitutions, and expeditions actually work in the field. I created a separate AI orientation document with 24 non-negotiable rules that every AI session must follow — from "the PC writes only recipes" to "never mock the database in tests" to "every form layout must use the width-expanding helper."

The application was verified across 32 dedicated testing sessions, each targeting a specific class of failure: a 1,000-iteration random-action fuzzer, cross-tab communication regression, end-to-end button click-throughs with real dialog windows, visual rendering checks, combinatorial state coverage across all 7 tabs, boundary input testing (negative numbers, impossible values, injection attacks), and form layout validation. The result: 1,637 behavioural tests that confirm the system works correctly under normal use, extreme inputs, and randomized chaos.

Safety Architecture — Recipe-Only Handshake

The PC writes only recipes to the controller. Start, stop, and all physical controls remain with the operator at the touchscreen.

Recipe Delivery
1 PC → PLC Write recipe targets into bounded register block
2 PC → PLC Set recipe-ready handshake signal
3 PLC → PC Controller validates and acknowledges receipt
4 PLC → PC Readback verification — values match within tolerance
5 PC → PLC Clear handshake — recipe delivered successfully
Operator Action on Touchscreen
6 Operator Reviews recipe on touchscreen · adjusts water for humidity · presses START
Production Monitoring
7 PLC → PC Poll batch status and controller mode (auto / manual / emergency)
8 PLC → PC Batch complete — record results, deduct inventory, generate certificate
Safety Gate Every write is checked against the allowed recipe range. Out-of-range writes are blocked and logged. The PC never sends start, stop, discharge, skip, or reset commands.

How I work
